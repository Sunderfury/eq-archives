Interesting!

I've been having a look at the AO protocols myself, and I'm finding that the playfield and loginhandler protocols have much in common with the chat protocol.

Instead of hooking into the winsock dll, I've inserted my own program between the AO client and the server by changing the address the AO client connects to by editing the dimensions.txt file (in cd_image\data\launcer) to connect to localhost. This file is replaced by the client each time it is started, but by commenting out the lines in DimensionServer (same dir.), it won't find it and use the old one.

Stats for the loginhandler and playfield protocols I've bothered to find is:
* Message length is an unsigned 16 bit integer in byte 5 and 6, but unlike the chat protocol, this is not the real length of the message - it is always padded to have a length of some divisible of 4. I.e. if the reported length is 14, real length is 16.
* Message ID is at bytes 10 and 11.

I haven't bothered decrypting many messages, only enough to have my program intercept and change incoming ip-adresses. These are:
* Logon from loginhandler: Message id 23, ip address as bytes 16-19, port at 20-21.
* Playshift (zoning) in playfield: Message id 60, ip address at bytes 12-15, port at 20-21.

Currently, my small prog. only recognize these messages, change ip addresses to localhost and spawn a new "spy process" between client and server whenever such a message is recieved.

I don't really know if this method is preferably in any way to hooking the winsock. Didn't think of it at the time, so I didn't look into it.

I would, however, be very interested to see how I could access information from the resource database. If someone could forward me the code to do that, I can think of several useful apps to look at . My mail is seefellow@hotmail.com .
