One thing that worries me about offset hacking is how easy it would be for verant to *legally* see that you're doing it. That's why I have only experimented on a throw-away account i have. They simply have to have some code clientside that compares a checksum of the spell area of process memory (the one we're changing). If this doesn't match they can take necessary action. I don't want to scare people, I think this research is cool , but please be aware.

On to my idea. This is also not undetectable (virtually nothing is), but would be much harder. I have been thinking about experimenting with injecting packets into the datastream sent to the client (messing with those sent to the server is easier to detect). It's encrypted, but it's using symmetric encryption with a session key that only changes when you zone. Once this key is known (ala SEQ method), it should be possible to encode data that the client can read. I'm sure there are other issues - such as sequence numbers etc. that could cause problems, but this could be another way we can hack the game.

I would imagine that anything that can be hacked via an in-memory hack can be effected using this method, but I'll come back to you when I have found out more.

BTW I am coding the program in Linux to run on my SEQ laptop, I prefer LINUX and it's easier to spoof the UDP packet header on it.

Has anyone else done/is doing work along this line? If so drop me an email at s_phreak@hotmail.com and perhaps we can compare notes.

-Speed Phreak
