    Re: Ubuntu Unity

    It would be cool if an editor/environment came along that was open source and started out with developing a framework similar to how emacs is designed in mind instead of just being a text editor or an IDE for a handful of languages and provided an interface for extending that framework in whatever way someone wanted such that the framework could hook into other tools in an exceptional way or be an e-mail client or irc client even as examples of its extendability. SL2 is or possibly could be such but I am not comfortable with it being proprietary and all of the user contributions being for naught at some point and it seems to be coming from the point of being a text editor first instead of being a platform that anything could be built upon. In my mind the program that did such a thing would be an improvement over its ancestor in the same way that vim takes vi and takes it into a broader direction.

    I guess xemacs was sort of like this but it is dead now. Programs that had similar goals have yet to gain traction and an enthusiastic user base. The main thing Emacs suffers (or benefits from, it's suffer imo) is how many ways the same thing can be achieved and low discoverability. Take the package manager for example, a default feature in emacs 24, however the sources it uses can be very different and some plugins are only available on some and not others or certain versions being only available on some or the latest version not being in any of those sources. I expect this to be transitioned away from since it is a new feature and more people will contribute to the default source instead of the previously used ones but it works as an example. One way this really turns up is when it comes to editing the .emacs file.

    Doing so (and understanding what is going on) requires a bit of elisp aptitude OR using the gui customizations interface which is very handicapped in what it shows is possible, which is completely insane. VIM (as an example) has its own language (vimscript) but editing the .vimrc and making useful changes is pretty straight forward after you see a few examples instead of requiring someone to read information on the 30 ways to change a keybinding when they just want to change the keybinding instead of reading a manifesto on it. Such things would have been avoided if legacy methods were deprecated and not even attempted to be supported version to version. Sure, some plugins would be broken in the process every major release but it would it greatly curb the amount of obfuscation in achieving something and the new way would be the default way instead of people unwittingly using a deprecated method. I suppose I just wish emacs would take a page from Apple/MS (for instance) and frequently kill off support instead of having 40 years of development and methods around so that it would be less intimidating and essentially impossible to write about with brevity.

    Re: Ubuntu Unity

    Evil-mode gained some really powerful new features in recent updates in the form of more powerful key-translations. This made accessing emacs keybindings in modes like ORG etc. much faster and more VIM like in addition to requiring a much lower barrier to entry for those familiar with the modes bindings. By merely typing cc or cx in normal state the C-c and C-x bindings are accessed while still retaining VIM's normal c verb purpose.

    It is mentioned in the key-translation section of the wiki:
    http://www.emacswiki.org/emacs/Evil

    I expect the popularity of evil to surge ever further now that those familiar with emacs commands accessed through C-x/C-c/M-x can drop into evil with no need to micro-optimize for each mode used. Emacs is now even more state of the art for anything that can deal with text. Luckily for Emacs veterans VIM concepts are trivial to grok in comparison and require much less muscle memory. Unfortunately for VIM users Emacs is nontrivial to gain proficiency in and not really useful outside of the uses you immediately have, e.g. learning for its own sake has a much lower pay off and will be largely wasted in the context of Emacs. Although learning Emacs lisp is well worth it and kind of required to some extent. Customization has a much higher barrier to entry than VIM does, but in the context of a tool that once used is useful for life that is not so bad.

    Language researchers and tool chain developers should emulate the models Haskell uses for wider adoption in plugins so that it requires less hacks to interact with efficiently and plugin development is trivial.

    edit: actually, due to a bug in evil currently that set up does break c as a verb due to a bug in operating pending mode. Still, will be great once that is fixed. the change verb is pretty powerful so I won't be making that trade off currently. There are other goodies on the page though.

    I think it is interesting that Emacs seems to always tie or beat VIM at VIM golf challenges, even though VIM has the reputation of being better for such editing tasks. The Emacs solutions never use external packages either, but to be fair Emacs ships with a ton of specialty packages by default.

    Also, Auctex + Preview + Predictive is the most pita thing to set up for some reason. Some emacs downloads bundle Auctex and Preview already thus avoiding the necessity of doing it yourself, but Predictive isn't included in any of them afaik. It's lucky that it is such a time saver once set up, but it is such a barrier to entry that it neuters an argument for utilizing it for the typical latex user. The predictive version listed on the repositories is like version .19 and the latest is .28 I believe, so it is super out of date in the repository.

    Such situations which kind of makes VIM's package managing situation (they require manual editing of the location to draw from, but can just pull from wherever without external work from the plugin dev) seem much more intelligent than expecting all the emacs devs to upload to a remote repository that they weren't already using and making a package that is formatted in a certain way.
