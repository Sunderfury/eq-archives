    Re: Lisp

    Originally Posted by MI Redeux
    I have a question for those of you with a deeper CS knowledge. I keep reading that having a garbage collector instead of managing memory manually results in a non-trivial performance cost, which I will accept on faith, but why is this the case? Is it just that the current algorithms are not intelligent enough to do whatever the programmer would have done or what? Why is doing it yourself potentially better performance wise?
    The computer must use resources to determine what memory can be reclaimed, in addition to actually reclaiming it. The two primary methods are tracing or reference-counting. With tracing, the garbage collector attempts to determine which memory is reachable and which isn't at the time it is invoked, and reclaims all unused space. With reference-counting, counters for all variables must be kept which keep track of how many of those variables reference a particular place in memory, and as soon as the counter hits zero the memory is reclaimed. Both of these techniques introduce runtime penalties and typically also are implemented in a more complex language that has other, unrelated overhead. (Though not always: ocaml and digital mars D for example)

    The runtime usually decides when to do the garbage collection. Very often in performance sensitive programs, there are particular times when resources are at a premium and other moments when resources are cheap. Obviously, if you don't actually need to run garbage collection during the critical sections you probably shouldn't, and languages/runtimes won't always get the right answer for this by themselves. (Some languages allow you to control the GC or turn it off)

    If a core is being consumed by garbage collection, it isn't doing more useful work. If a program is written such that it can only effectively use one core, wouldn't you rather have the other cores in a low-power state rather than running the garbage collector?     Re: Lisp

    Originally Posted by Salamane
    If a core is being consumed by garbage collection, it isn't doing more useful work. If a program is written such that it can only effectively use one core, wouldn't you rather have the other cores in a low-power state rather than running the garbage collector? You are also overlooking how poorly GC interacts with most virtual memory schemes. It isn't a big win to have to keep pulling in pages to look for space that can be reclaimed.

    If you use Objective-C, ARC (automatic reference counting) pretty much attains what you are going for (simple memory management, low impact at run time, the compiler does most of the work).
    So, .NET uses a three-generation reference system for doing garbage collection. But I think it's worth talking first about how "multi-tasking" actually works in a computer, without taking into consideration multiple-cores.

    Let's say you have a program that you've written that does two basic functions, one function is calculating data, and the other is displaying it. Since you're an advanced programmer, you spin up a second "thread" to handle the second function. These two threads are then entered into the system thread manager, and in turn, each thread is serviced. Some threads can raise their own priority to be serviced sooner or more frequently than other threads, but that's not really all that important, because when you get down to it, your computer is running hundreds, and usually thousands of threads at any given time. And they're all being serviced in turn by the scheduler, but they're not actually running at the same time.

    So, yeah, you could probably write a slightly better GC, but there are fewer and fewer reasons to do so on modern hardware. The GC in Windows has been in use in the system for a decade and has been optimized in ways that you're probably not likely to understand, as they're probably specific to the way Windows works under the hood, so the argument that you're not likely to be able to write a better GC is probably true; and moreso, it's highly unlikely you'd be able to do it in such a way that someone would think it was a good use of your time as a developer!

    Re: Lisp

    In hindsight, I am surprised that the AI community moved away from using LISP as its primary tool. It is mainly an area dominated by academics, a population which is very protected from industry trends and has greater autonomy than that present in industry, so they are able to choose whatever language they wished to use. Perhaps contributing factors were the decline of improvement in AI research until the 90s and LISP no longer being a language used to teach newbies. In the mean time from the 70s to 90s the people who were familiar with LISP in AI were dying off or retiring leaving the younger population free to adopt new tools. As a result less popular languages gained usage among a population that is not necessarily formally trained in CS so prefer languages more suited for statistics and rapid prototyping as well as accessibility such as OCaml, R, Perl, Python, Haskell, Julia, and Matlab/Octave. That is my hypothesis anyways.
